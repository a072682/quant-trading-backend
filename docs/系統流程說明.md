# 量化交易系統 — 後端流程說明

> 本文件說明整個後端系統的完整運作流程，適合完全不懂程式的讀者閱讀。

---

## 一、系統啟動流程

當你執行 `uvicorn app.main:app` 啟動伺服器時，程式會依序做以下四件事：

**步驟 1：[main.py] FastAPI app 建立**
→ 建立一個名為「量化交易系統 API」的網路服務應用程式
→ 根據 `.env` 設定，決定是否開放 `/docs` 文件頁面（開發環境才開）
→ 將 `lifespan` 函式綁定，告訴程式「啟動時和關閉時要做什麼」

**步驟 2：[main.py] lifespan 執行 — 資料表初始化**
→ 連線到 PostgreSQL 資料庫
→ 自動檢查並建立四張資料表：`users`、`stock_pool`、`filter_status`、`signals`
→ 若資料表已存在則跳過，不會清空舊資料
→ 印出「資料表初始化完成」，代表啟動成功

**步驟 3：[main.py] CORS 設定**
→ 設定哪些前端網域可以呼叫後端 API（從 `.env` 的 `ALLOWED_ORIGINS` 讀取）
→ 沒有這個設定，瀏覽器會因安全限制拒絕前端的跨網域請求

**步驟 4：[main.py → router.py] router 掛載**
→ 將所有 API 端點統一掛載，所有路徑以 `/api` 開頭
→ 底下分為四個群組：`/api/health`、`/api/auth`、`/api/stocks`、`/api/signals`

## 二、連線確認流程

**起點：前端發出 `GET /api/health`**

步驟 1：[main.py] FastAPI 框架
→ 收到請求，根據路徑 `/api/health` 找到對應的處理函式
→ 將請求交給 `health.py` 的 `health_check()` 函式

步驟 2：[health.py] `health_check()`
→ 收到請求，呼叫 `check_database()` 確認資料庫是否連線正常
→ 傳給 `health_service.py` 的 `check_database()`

步驟 3：[health_service.py] `check_database()`
→ 向資料庫連線池借出一條連線
→ 執行最簡單的 SQL 指令 `SELECT 1`（只是確認連線通了，不查任何資料）
→ 成功回傳 `"ok"`，失敗回傳 `"error"`
→ 將結果回傳給 `health_check()`

步驟 4：[health.py] `health_check()`（繼續）
→ 根據回傳的狀態，組合回應內容
→ 回傳給前端

**終點：前端收到以下格式的回應（HTTP 200）**
```json
{
  "success": true,
  "server": "ok",
  "database": "ok",
  "timestamp": "2026-04-29T10:00:00.000000+00:00"
}
```

## 三、會員系統流程

### 3-1 註冊流程

**起點：前端發出 `POST /api/auth/register`，帶著 email、username、password**

步驟 1：[auth.py] `register()`
→ 收到請求，由 Pydantic 自動驗證 email 格式、欄位是否齊全
→ 呼叫 `check_email_exists()` 確認 email 是否已被使用

步驟 2：[auth_service.py] `check_email_exists()`
→ 內部呼叫 `get_user_by_email()` 查詢資料庫
→ 找到資料 → 回傳 `True`（email 已存在）
→ 找不到資料 → 回傳 `False`（可以註冊）

步驟 3：[auth.py] `register()`（繼續）
→ 若 `True`：拋出 HTTP 400 錯誤，訊息「此 Email 已被註冊」，流程結束
→ 若 `False`：呼叫 `hash_password()` 將明碼加密

步驟 4：[security.py] `hash_password()`
→ 使用 bcrypt 演算法，加入隨機鹽值對密碼加密
→ 相同密碼每次加密結果都不同，確保安全
→ 回傳加密後的密碼字串（格式如 `$2b$12$xxx...`）

步驟 5：[auth_service.py] `create_user()`
→ 建立 User 物件（email、username、加密後密碼）
→ 寫入 `users` 資料表
→ 重新從資料庫載入，取得自動產生的 `id`
→ 回傳完整的 User 物件

步驟 6：[security.py] `create_access_token()`
→ 以使用者 `id` 為識別碼，產生 JWT Token
→ Token 內含到期時間（預設 7 天）
→ 以 `SECRET_KEY` 簽名，確保無法偽造

步驟 7：[auth.py] `register()`（回傳）
→ 組合回應，帶入 Token

**終點：前端收到以下格式的回應（HTTP 201 Created）**
```json
{
  "message": "註冊成功",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiJ9...",
    "token_type": "bearer"
  }
}
```

---

### 3-2 登入流程

**起點：前端發出 `POST /api/auth/login`，帶著 username（填入 email）、password**

步驟 1：[auth.py] `login()`
→ 收到請求，由 FastAPI 內建的 `OAuth2PasswordRequestForm` 解析 username 和 password
→ 呼叫 `get_user_by_email()` 用 email 查詢使用者

步驟 2：[auth_service.py] `get_user_by_email()`
→ 對 `users` 資料表執行 `SELECT WHERE email = ?`
→ 找到 → 回傳 User 物件
→ 找不到 → 回傳 `None`

步驟 3：[auth.py] `login()`（繼續）
→ 若找不到使用者：拋出 HTTP 401 錯誤，訊息「帳號或密碼錯誤」，流程結束
→ 若找到：呼叫 `verify_password()` 驗證密碼

步驟 4：[security.py] `verify_password()`
→ 用 bcrypt 將輸入的明碼與資料庫的加密密碼比對
→ 相符 → 回傳 `True`
→ 不相符 → 回傳 `False`

步驟 5：[auth.py] `login()`（繼續）
→ 若 `False`：拋出 HTTP 401 錯誤，流程結束
→ 若 `True`：呼叫 `create_access_token()` 產生 Token

步驟 6：[security.py] `create_access_token()`
→ 以使用者 `id` 為識別碼，產生 JWT Token（同註冊流程）
→ 回傳 Token 字串

步驟 7：[auth.py] `login()`（回傳）
→ 組合回應，帶入 Token

**終點：前端收到以下格式的回應（HTTP 200）**
```json
{
  "message": "登入成功",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiJ9...",
    "token_type": "bearer"
  }
}
```

### 3-3 登出流程

> 本系統使用 JWT Token，後端不儲存 Token。
> 登出的本質是「確認你真的有登入」，然後告訴前端「可以清除 Token 了」。

**起點：前端發出 `POST /api/auth/logout`，在 Header 帶著 `Authorization: Bearer <token>`**

步驟 1：[security.py] `get_current_user()`（依賴注入，在進入函式前自動執行）
→ 從 Header 取出 Token 字串
→ 呼叫 `decode_token()` 解讀並驗證 Token

步驟 2：[security.py] `decode_token()`
→ 用 `SECRET_KEY` 驗證 Token 簽名是否正確
→ 確認 Token 是否在有效期內
→ 有效 → 取出使用者 `id` 回傳
→ 無效或過期 → 回傳 `None`

步驟 3：[security.py] `get_current_user()`（繼續）
→ 若 `None`：拋出 HTTP 401 錯誤，流程結束
→ 若有效：回傳使用者 `id`，讓 `logout()` 繼續執行

步驟 4：[auth.py] `logout()`
→ 確認身份後，組合登出成功的回應（後端本身不做任何清除動作）
→ 前端收到後自行刪除本地儲存的 Token

**終點：前端收到以下格式的回應（HTTP 200），前端自行清除 Token**
```json
{
  "message": "登出成功",
  "data": {
    "logged_out": true,
    "timestamp": "2026-04-29T10:00:00.000000+00:00"
  }
}
```

## 四、股票池建立流程

> 股票池是指「通過基本條件篩選、值得進一步評分的股票清單」。
> 篩選條件：殖利率、市值等基本面指標。
> 所有 `/stocks` 端點都需要在 Header 帶著有效的 JWT Token。

### 4-1 觸發篩選

**起點：前端發出 `POST /api/stocks/filter`，Header 帶著 Token**

步驟 1：[security.py] `get_current_user()`（自動執行）
→ 驗證 Token 是否有效，無效則回傳 401，流程結束

步驟 2：[stocks.py] `trigger_filter()`
→ 查詢 `filter_status` 資料表，確認是否已有 `status = "running"` 的任務
→ 若有：拋出 HTTP 400 錯誤，訊息「篩選任務執行中，請稍後再試」，流程結束
→ 若無：建立新的 FilterStatus 紀錄（`status = "running"`，記錄開始時間）
→ 將 `_run_filter_task()` 加入背景任務佇列
→ 立即回傳 HTTP 202，不等待背景任務完成

步驟 3：（背景執行）[stocks.py] `_run_filter_task()`
→ 建立獨立的資料庫連線（不可重用 request 的連線）

  步驟 3-1：[twse_client.py] `fetch_twse_stocks()`
  → 向台灣證券交易所（TWSE）官網抓取所有上市股票清單
  → 回傳原始股票清單（含代號、名稱）

  步驟 3-2：[yfinance_client.py] `filter_by_yfinance()`
  → 逐批查詢 yfinance，取得每支股票的殖利率和市值
  → 依設定的篩選條件過濾
  → 回傳通過篩選的股票清單（含代號、名稱、殖利率、市值）

  步驟 3-3：[stocks_service.py] `save_stock_pool()`
  → 先刪除 `stock_pool` 資料表的所有舊資料
  → 批次寫入本次通過篩選的新股票
  → 提交至資料庫

→ 篩選成功：更新 FilterStatus 為 `completed`，記錄完成時間和通過數量
→ 篩選失敗：更新 FilterStatus 為 `failed`，記錄錯誤原因（截斷至 500 字）

**終點：前端立即收到（HTTP 202 Accepted），背景任務繼續執行**
```json
{
  "message": "篩選任務已啟動，請稍後透過 GET /status 查詢進度"
}
```

### 4-2 查詢股票池

**起點：前端發出 `GET /api/stocks/pool`，Header 帶著 Token**

步驟 1：[security.py] `get_current_user()`（自動執行）
→ 驗證 Token，無效則回傳 401

步驟 2：[stocks.py] `get_pool()`
→ 呼叫 `get_stock_pool()` 查詢資料庫

步驟 3：[stocks_service.py] `get_stock_pool()`
→ 對 `stock_pool` 資料表執行 `SELECT *`
→ 回傳所有 StockPool 物件清單

步驟 4：[stocks.py] `get_pool()`（繼續）
→ 將每個 StockPool ORM 物件轉換為 API 回傳格式（StockPoolItem）
→ 組合回應

**終點：前端收到以下格式的回應（HTTP 200）**
```json
{
  "message": "取得 50 支股票",
  "data": [
    {
      "stock_code": "2330",
      "stock_name": "台積電",
      "yield_pct": 2.1,
      "market_cap": 18000.0,
      "updated_at": "2026-04-29T10:00:00+00:00"
    }
  ]
}
```

### 4-3 查詢篩選狀態

**起點：前端發出 `GET /api/stocks/status`，Header 帶著 Token**

步驟 1：[security.py] `get_current_user()`（自動執行）
→ 驗證 Token，無效則回傳 401

步驟 2：[stocks.py] `get_status()`
→ 呼叫 `get_filter_status()` 查詢最新一筆狀態

步驟 3：[stocks_service.py] `get_filter_status()`
→ 對 `filter_status` 資料表，依 `started_at` 降冪排序取第一筆
→ 有資料回傳 FilterStatus 物件，無資料回傳 `None`

步驟 4：[stocks.py] `get_status()`（繼續）
→ 若 `None`：拋出 HTTP 404，訊息「尚未有篩選紀錄」
→ 若有資料：將 ORM 物件轉換為 FilterStatusOut 格式，組合回應

**終點：前端收到以下格式的回應（HTTP 200）**
```json
{
  "message": "查詢成功",
  "data": {
    "status": "completed",
    "started_at": "2026-04-29T10:00:00+00:00",
    "completed_at": "2026-04-29T10:05:00+00:00",
    "stock_count": 50,
    "error_message": null
  }
}
```

> **status 可能的值：**
> - `running`：篩選任務正在執行中
> - `completed`：篩選完成
> - `failed`：篩選失敗（error_message 會有錯誤原因）

## 五、評分系統流程

> 評分系統會對股票池裡的每支股票進行多維度評分，並呼叫 Claude AI 分析前幾名股票。
> 所有 `/signals` 端點都需要在 Header 帶著有效的 JWT Token。

### 5-1 觸發評分

**起點：前端發出 `POST /api/signals/run`，Header 帶著 Token**

步驟 1：[security.py] `get_current_user()`（自動執行）
→ 驗證 Token，無效則回傳 401

步驟 2：[signals.py] `run_scoring()`
→ 檢查模組層級旗標 `_is_running` 是否為 `True`
→ 若 `True`：拋出 HTTP 400 錯誤，訊息「評分任務執行中，請稍後再試」，流程結束
→ 若 `False`：將 `_run_scoring_task()` 加入背景任務佇列
→ 立即回傳 HTTP 202，不等待背景任務完成

步驟 3：（背景執行）[signals.py] `_run_scoring_task()`
→ 將 `_is_running` 設為 `True`，防止重複觸發
→ 建立獨立的資料庫連線

  步驟 3-1：[calculator.py] `fetch_market_score()`
  → 抓取大盤（台灣加權指數）的狀態
  → 計算大盤情緒分數（所有股票共用這一個分數，只抓一次）
  → 回傳整數分數

  步驟 3-2：對 `stock_pool` 資料表執行 `SELECT *`
  → 取得所有待評分的股票清單

  步驟 3-3：逐支股票執行評分（對每支股票重複以下動作）

    [calculator.py] `calculate_total_score()`
    → 抓取該股票的法人買賣超、均線位置、成交量比率、殖利率等資料
    → 計算各分項得分（institutional / ma / volume / yield / futures）
    → 加總為 total_score
    → 回傳分數字典

    [signals_service.py] `save_signal()`
    → 查詢今日是否已有該股票的評分記錄
    → 有 → 更新現有記錄的分項分數
    → 沒有 → 新增一筆記錄
    → 提交至資料庫

  步驟 3-4：[signals_service.py] `get_top_signals()`
  → 取出今日總分超過門檻（預設 4 分）的前幾名股票

  步驟 3-5：對每支高分股票執行 AI 分析

    [ai_analyzer.py] `analyze_signal()`
    → 將股票代號、名稱及各分項分數整理成說明文字
    → 呼叫 Claude AI（Anthropic API）分析
    → AI 回傳三個欄位：
      - `ai_action`：建議動作（buy / watch / sell）
      - `ai_reason`：文字說明（AI 的分析理由）
      - `confidence`：信心值（0 到 100 的整數）

    更新 `signals` 資料表對應記錄的 AI 分析欄位
    → 提交至資料庫

→ 全部完成後，將 `_is_running` 設回 `False`

**終點：前端立即收到（HTTP 202 Accepted），背景任務繼續執行**
```json
{
  "message": "評分任務已啟動，請稍後透過 GET /today 查詢結果"
}
```

## 六、資料表說明

### `users` — 使用者資料表

儲存所有註冊帳號的基本資料。

| 欄位 | 資料類型 | 說明 |
|---|---|---|
| `id` | 字串（UUID） | 唯一識別碼，自動產生 |
| `email` | 字串（255字） | 電子郵件，唯一、不可重複 |
| `username` | 字串（100字） | 使用者顯示名稱 |
| `hashed_password` | 字串（255字） | bcrypt 加密後的密碼（不儲存明碼） |
| `is_active` | 布林值 | 帳號是否啟用，預設 `true` |
| `created_at` | 時間 | 帳號建立時間（UTC 時區） |

---

### `stock_pool` — 股票池資料表

儲存本次篩選通過的股票清單。每次篩選前會**清空舊資料**再寫入新資料。

| 欄位 | 資料類型 | 說明 |
|---|---|---|
| `id` | 字串（UUID） | 唯一識別碼，自動產生 |
| `stock_code` | 字串（10字） | 股票代碼（如 `2330`） |
| `stock_name` | 字串（50字） | 股票名稱（如 `台積電`） |
| `yield_pct` | 浮點數 | 殖利率（百分比，如 `4.07`） |
| `market_cap` | 浮點數 | 市值（億台幣） |
| `updated_at` | 時間 | 寫入時間（UTC 時區） |

---

### `filter_status` — 篩選任務狀態表

每次觸發篩選時建立一筆紀錄，追蹤任務的執行進度。**不會清空**，保留所有歷史紀錄。

| 欄位 | 資料類型 | 說明 |
|---|---|---|
| `id` | 字串（UUID） | 唯一識別碼，自動產生 |
| `status` | 字串（20字） | 任務狀態：`running` / `completed` / `failed` |
| `started_at` | 時間 | 任務開始時間（UTC 時區） |
| `completed_at` | 時間 | 任務完成或失敗時間（UTC 時區） |
| `stock_count` | 整數 | 本次通過篩選的股票數量（完成後才有值） |
| `error_message` | 文字 | 失敗原因（失敗時才有值，最多 500 字） |

---

### `signals` — 每日評分訊號資料表

儲存每天每支股票的評分結果與 AI 分析。同一支股票當日重複評分會**更新**而非新增。

| 欄位 | 資料類型 | 說明 |
|---|---|---|
| `id` | 字串（UUID） | 唯一識別碼，自動產生 |
| `date` | 字串（10字） | 評分日期（台灣時區，格式 `YYYY-MM-DD`） |
| `stock_code` | 字串（10字） | 股票代碼 |
| `stock_name` | 字串（50字） | 股票名稱 |
| `institutional_score` | 整數 | 法人買賣超得分 |
| `ma_score` | 整數 | 均線位置得分 |
| `volume_score` | 整數 | 成交量量比得分 |
| `yield_score` | 整數 | 殖利率得分 |
| `futures_score` | 整數 | 大盤情緒得分 |
| `total_score` | 整數 | 各分項加總總分 |
| `ai_action` | 字串（10字） | AI 建議動作：`buy` / `watch` / `sell` |
| `ai_reason` | 長文字 | AI 分析說明文字 |
| `confidence` | 整數 | AI 信心值（0 到 100） |
| `created_at` | 時間 | 首次建立時間（UTC 時區） |
| `updated_at` | 時間 | 最後更新時間（UTC 時區） |

## 七、目前開放的 API 清單

| 方法 | 路徑 | 說明 | 是否需要登入 |
|---|---|---|---|
| `GET` | `/api/health` | 確認伺服器與資料庫連線狀態 | 否 |
| `POST` | `/api/auth/register` | 使用者註冊 | 否 |
| `POST` | `/api/auth/login` | 使用者登入，取得 Token | 否 |
| `POST` | `/api/auth/logout` | 使用者登出 | 是 |
| `POST` | `/api/stocks/filter` | 觸發股票池篩選（背景執行） | 是 |
| `GET` | `/api/stocks/pool` | 取得股票池清單 | 是 |
| `GET` | `/api/stocks/status` | 查詢篩選任務狀態 | 是 |
| `POST` | `/api/signals/run` | 觸發評分任務（背景執行） | 是 |
| `GET` | `/api/signals/today` | 取得今日所有評分 | 是 |
| `GET` | `/api/signals/top` | 取得今日 AI 推薦股票（前 3 名） | 是 |

> **使用需要登入的 API 時**，請在 HTTP Header 加入：
> `Authorization: Bearer <你的 Token>`
