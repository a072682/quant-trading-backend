# 系統架構說明

## 一、整體架構圖

```
┌─────────────────────────────────────────────────────────┐
│                      前端（React）                        │
│  儀表板 / 股票池 / 模擬交易 / K線圖 / 推薦清單             │
└────────────────────┬────────────────────────────────────┘
                     │  HTTP / WebSocket
                     ▼
┌─────────────────────────────────────────────────────────┐
│               後端 API（FastAPI）                         │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐  │
│  │  認證模組 │  │ 評分模組 │  │ 股票池   │  │ 交易   │  │
│  │  /auth   │  │ /signals │  │ /stocks  │  │/trades │  │
│  └──────────┘  └──────────┘  └──────────┘  └────────┘  │
│                                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐  │
│  │ 模擬交易 │  │ 帳戶持倉 │  │  排程器（APScheduler）│  │
│  │/simulation│  │/positions│  │ 每日 14:00 / 每週一  │  │
│  └──────────┘  └──────────┘  └──────────────────────┘  │
└──────┬──────────────┬──────────────┬────────────────────┘
       │              │              │
       ▼              ▼              ▼
┌────────────┐  ┌──────────┐  ┌──────────────────────────┐
│ PostgreSQL │  │ 富果 API │  │       外部資料來源         │
│ 資料庫     │  │（下單）  │  │  yfinance（股價/K線）     │
│            │  └──────────┘  │  TWSE API（法人買賣超）   │
│ signals    │                │  Anthropic Claude（AI分析）│
│ trades     │                └──────────────────────────┘
│ simulation │
│ stock_pool │
│ filter_stat│
│ users      │
└────────────┘
```

**簡單說明**：
- 前端透過 HTTP 呼叫後端 API，後端再去抓各種外部資料（股價、法人資料、AI）
- 資料都存在 PostgreSQL 資料庫，前端查詢時直接從資料庫撈
- 排程器像一個定時鬧鐘，在固定時間自動執行評分任務，不需要人工觸發

## 二、資料表說明

### 1. `users`（使用者帳號表）

| 欄位名稱 | 資料類型 | 說明 |
|---------|---------|------|
| id | UUID | 每個使用者的唯一識別碼（自動產生） |
| email | 文字 | 登入用的電子郵件 |
| username | 文字 | 顯示名稱 |
| hashed_password | 文字 | 加密後的密碼（不儲存明碼） |
| is_active | 布林 | 帳號是否啟用中 |
| created_at | 時間 | 帳號建立時間 |

**用途**：管理系統登入帳號。所有 API 都需要先登入才能使用。

---

### 2. `signals`（每日評分訊號表）

這是系統最核心的資料表，儲存每天每一檔股票的評分結果。

| 欄位名稱 | 資料類型 | 說明 |
|---------|---------|------|
| id | UUID | 唯一識別碼 |
| date | 文字 | 評分日期，格式 YYYY-MM-DD（例：2026-04-27） |
| stock_code | 文字 | 股票代號（例：2886） |
| stock_name | 文字 | 股票名稱（例：兆豐金） |
| institutional_score | 整數 | 法人買賣超得分（-2 到 +2） |
| ma_score | 整數 | 均線位置得分（-1 到 +2） |
| volume_score | 整數 | 成交量得分（-1 到 +2） |
| yield_score | 整數 | 殖利率得分（-1 到 +2） |
| futures_score | 整數 | 台指期市場情緒得分（-3 到 +3） |
| total_score | 整數 | 五項分數加總（理論最高 +11，最低 -7） |
| ai_action | 文字 | AI 建議動作：buy（買進）/ watch（觀察）/ sell（賣出） |
| ai_reason | 長文字 | AI 給出的分析理由（50 字內的白話說明） |
| created_at | 時間 | 記錄建立時間 |
| updated_at | 時間 | 最後更新時間（用來判斷是否需要重算） |

**用途**：前端儀表板顯示今日推薦、歷史評分走勢，以及 AI 建議，都從這張表撈資料。

---

### 3. `trades`（實際交易紀錄表）

| 欄位名稱 | 資料類型 | 說明 |
|---------|---------|------|
| id | UUID | 唯一識別碼 |
| date | 文字 | 交易日期 |
| stock_code | 文字 | 股票代號 |
| stock_name | 文字 | 股票名稱 |
| action | 文字 | 買或賣：buy / sell |
| price | 小數 | 成交價格（元） |
| shares | 整數 | 成交股數 |
| total_amount | 小數 | 總金額（價格 × 股數） |
| profit | 小數 | 損益金額（賣出時才有值，買入為 0） |
| reason | 長文字 | 交易理由說明 |
| order_id | 文字 | 富果 API 回傳的委託單號（用來對帳） |
| created_at | 時間 | 紀錄建立時間 |

**用途**：記錄真實下單的買賣紀錄，可以查詢本月損益統計。

---

### 4. `simulation_trades`（模擬交易紀錄表）

| 欄位名稱 | 資料類型 | 說明 |
|---------|---------|------|
| id | UUID | 唯一識別碼 |
| date | 文字 | 交易日期 |
| stock_code | 文字 | 股票代號 |
| stock_name | 文字 | 股票名稱 |
| action | 文字 | buy（買入）或 sell（賣出） |
| price | 小數 | 模擬成交價格 |
| shares | 整數 | 模擬股數 |
| total_amount | 小數 | 模擬總金額 |
| signal_score | 整數 | 買入當時的評分總分（記錄用） |
| status | 文字 | open（持倉中）/ closed（已平倉） |
| buy_price | 小數 | 買入價格（用來計算損益） |
| profit | 小數 | 損益金額（平倉時計算） |
| profit_pct | 小數 | 損益百分比（例：6.5 代表 +6.5%） |
| created_at | 時間 | 紀錄建立時間 |

**用途**：模擬「紙上交易」，不動真錢，用來驗證評分策略的準確度。系統會自動在達到 +6% 停利或 -3% 停損時平倉。

---

### 5. `stock_pool`（股票池表）

| 欄位名稱 | 資料類型 | 說明 |
|---------|---------|------|
| id | UUID | 唯一識別碼 |
| stock_code | 文字 | 股票代號 |
| stock_name | 文字 | 股票名稱 |
| yield_pct | 小數 | 殖利率（%），例：5.2 代表 5.2% |
| market_cap | 小數 | 市值（億元） |
| updated_at | 時間 | 最後更新時間 |

**用途**：儲存「值得追蹤」的股票名單。每週一自動從台灣證交所篩選一次，符合條件的股票才會進入這個池子，之後的評分都以這個池子為範圍。

---

### 6. `filter_status`（篩選任務狀態表）

| 欄位名稱 | 資料類型 | 說明 |
|---------|---------|------|
| id | UUID | 唯一識別碼 |
| status | 文字 | 任務狀態：idle（閒置）/ running（執行中）/ completed（完成）/ failed（失敗） |
| started_at | 時間 | 任務開始時間 |
| completed_at | 時間 | 任務完成時間 |
| stock_count | 整數 | 本次篩選後的股票數量 |
| error_message | 長文字 | 若失敗，記錄錯誤原因（最多 500 字） |

**用途**：因為股票篩選要跑很久（可能好幾分鐘），前端不能一直等，所以用這張表記錄進度。前端每隔幾秒查詢一次這張表，就能知道篩選有沒有完成。

## 三、資料表關聯

```
users
  └── （無直接外鍵，但 API 全部需要登入才能呼叫）

stock_pool
  └── 提供股票代號給 signals 使用
  └── 篩選狀態記錄在 filter_status

signals（核心）
  ├── 由股票池中的股票代號產生
  └── 總分 >= 6 的股票會觸發 AI 分析

simulation_trades
  └── 參考 signals.total_score 決定是否開倉
  └── status="open" 的股票會被排除在今日推薦之外

trades（實際交易）
  └── 獨立運作，不與其他表有直接關聯
```

**整體關係說明**：

1. **股票池** 是評分的「原料清單」，先篩好哪些股票值得追蹤
2. **評分表**（signals）是核心，記錄每天每檔股票的評分結果
3. **模擬交易表** 和 **實際交易表** 都是「下單紀錄」，前者是虛擬的、後者是真實的
4. **篩選狀態表** 是背景任務的「進度條」，讓前端知道篩選有沒有完成
